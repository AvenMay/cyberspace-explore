# Framework

## 一、 成熟度模型

这类框架用于评估组织当前的应用安全能力，并制定长期改进路线图。

- **OWASP SAMM (Software Assurance Maturity Model):**
  
  - **特点：** 开放、规范、可衡量。它将应用安全划分为 5 个业务功能（治理、设计、实现、验证、运营），每个功能下有 3 个实践，共 15 个安全实践。
    
  - **适用：** 想要从零建立 AppSec 体系并有量化指标的中小型到大型企业。
    
- **BSIMM (Building Security In Maturity Model):**
  
  - **特点：** “描述性”框架。它不是告诉你“应该怎么做”，而是通过观察数百家头部企业的实际做法，告诉你“别人是怎么做的”。
    
  - **适用：** 预算充足、希望与行业标杆（如金融、科技巨头）对标的大型组织。

## 二、 开发生命周期框架

这类框架侧重于将安全动作嵌入到软件开发的每一个阶段（即 DevSecOps 或 S-SDLC）。

- **Microsoft SDL (Security Development Lifecycle):**
  
  - **地位：** 行业鼻祖和事实标准。它定义了从需求、设计（威胁建模）、编码到发布、响应的完整安全阶段。
    
  - **核心：** 强调“左移”（Shift Left），在代码还没写出来之前就开始考虑安全。
    
- **NIST SSDF (Secure Software Development Framework - SP 800-218):**
  
  - **特点：** 美国联邦政府推行的标准。它不规定具体工具，而是规定“成果”，分为：准备组织、保护软件、生产安全软件、响应漏洞。
    
  - **趋势：** 在供应链安全和合规性要求极高的行业（如能源、军工、关键基础设施）是强制性的参考标准。

## 三、 技术验证与控制标准

这类框架更偏向技术底层，告诉开发和测试人员具体的漏洞检查点。

- **OWASP ASVS (Application Security Verification Standard):**
  
  - 这是目前最权威的应用层安全控制列表。它将安全要求分为三级（Level 1-3），从基础的 Web 应用到处理高度敏感数据的金融应用都有对应的清单。
- **OWASP Top 10:** * 虽然它是一个“漏洞排名”，但它被广泛当作入门级的 AppSec 检查准则。
